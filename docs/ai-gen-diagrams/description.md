## system-architecture.svg

System-level block diagram showing how the smart-speaker components connect and communicate.

**What it shows:**
- User-facing interfaces: voice, NFC, web dashboard, mobile app.
- ESP32 edge device with internal modules: microphone (I2S RX), speaker amplifier (I2S TX), NFC reader (SPI), OLED (I2C), wake-word runtime, audio streaming, playback.
- Voice backend pipeline: audio capture endpoint, WebSocket streaming, Silero VAD, Groq Whisper STT, single-turn ReAct agent with tool dispatch, VieNeu TTS, media transcoding.
- Database API: user profile, NFC identity, memory, preferences, alarms, timers, lists, media history, logs.
- External services: OpenRouter LLM, Groq STT, TTS provider, YouTube, web search.
- Data flow direction: PCM audio chunks upstream; control JSON + TTS/media URLs downstream.

**Key insight:** The ESP32 is an active edge node (wake-word detection + realtime I/O), not a passive microphone. The backend owns AI processing; the device owns realtime behavior.

## device-state-machine.svg

ESP32 behavioral state machine covering identity onboarding, the voice interaction loop, and recovery/override paths.

**What it shows:**
- Onboarding path: splash/startup → await NFC → load profile → register/error → greeting.
- Core voice loop: `wait_wakeword → streaming → thinking → speaking → wait_wakeword`.
- Backend-driven states: backend sends `wait_wakeword`, `streaming`, `thinking`, `speaking` commands.
- Recovery paths:
  - No-speech timeout: `streaming → wait_wakeword`.
  - Playback complete: `speaking → wait_wakeword`.
  - NFC interrupt: can interrupt `thinking`/`speaking` back to wake-word mode.
  - Wi-Fi reconnect: temporary overlay, resumes previous state.
  - Schedule alert: overrides active behavior, returns after dismiss.

**Key insight:** The critical invariant — when the UI shows sleep/wake-word screen, wake-word detection must be actively running. This ensures the speaker is always ready after every interaction.

## hardware-wiring.svg

Physical wiring diagram for the ESP32 DOIT DevKit V1 and its four peripheral modules.

**What it shows:**
- ESP32 DOIT DevKit V1 as the main controller.
- OLED SSD1306 on I2C: `GPIO21 (SDA)`, `GPIO22 (SCL)`, powered at 3.3V.
- RC522 NFC reader on SPI: `GPIO16 (SS)`, `GPIO17 (RST)`, `GPIO18 (SCK)`, `GPIO19 (MISO)`, `GPIO23 (MOSI)`, powered at 3.3V.
- INMP441 microphone on I2S RX: `GPIO26 (BCLK)`, `GPIO27 (WS)`, `GPIO32 (SD)`, powered at 3.3V.
- MAX98357A amplifier on I2S TX: `GPIO25 (DIN)`, shared `GPIO26 (BCLK)`, shared `GPIO27 (LRC)`, `GPIO13 (SD enable)`, powered at 5V VIN.
- All modules share common ground.
- Shared I2S clock lines between mic (RX) and amp (TX) require firmware to switch audio routes.

**Key insight:** Simple hardware — one ESP32, four modules, clear bus separation (I2C, SPI, I2S RX, I2S TX). The main constraint is stable I2S route switching since capture and playback share clock lines.

## wakeword-model-pipeline.svg

Data and model pipeline for the custom TFLite Micro wake-word detector.

**What it shows:**
- Data sources: positive wake-up clips, unknown speech commands, public noise datasets, real ESP32-recorded noise.
- Augmentation: every clip resampled to 1s @ 16kHz, mixed with random gain and SNR-based noise.
- Dataset: 15,000 balanced clips — 25% wake-up, 50% unknown, 25% noise. Train/validation/test split: 9,600 / 2,400 / 3,000.
- Model: MFCC feature extraction → quantized TFLite classifier → Edge Impulse C++ library.
- Runtime on ESP32: INMP441 capture → continuous inference → threshold + release + cooldown trigger logic.

**Key insight:** Wake-word detection is fully local — no server involvement until after wake-up. The model is trained to be conservative (reject ordinary speech and background noise).

## voice-sequence.svg

End-to-end sequence diagram for one complete voice interaction turn.

**What it shows:**
1. User speaks wake word.
2. ESP32 detects wake word locally (Edge Impulse TFLite model).
3. ESP32 calls `POST /api/audio/start` with capture token, WebSocket address, NFC tag.
4. Backend connects to ESP32 WebSocket.
5. ESP32 streams PCM packets (320 samples = 20ms @ 16kHz).
6. Backend runs Silero VAD for utterance detection and end-of-speech.
7. Backend transcribes with Groq Whisper STT.
8. Single-turn ReAct agent routes intent, calls tools if needed, composes answer.
9. Backend generates TTS, prepares `assistant_playback` JSON with optional media URL.
10. ESP32 plays TTS/media, returns to wake-word mode.

**Key insight:** The contract between ESP32 and backend stays simple — PCM audio upstream, control JSON + URLs downstream. Heavy AI work stays on the server.

## rtos-gantt.svg

Gantt-style timeline of FreeRTOS task execution across one full voice interaction cycle.

**What it shows:**
- System phases: boot → NFC login → wait wake-word → streaming → thinking → speaking → back to wake-word.
- RTOS tasks:
  - `uiTask`: state transitions, OLED updates, serial telemetry, RFID polling.
  - `audio_capture`: I2S microphone sampling, queue fan-out.
  - `wakeword`: continuous Edge Impulse inference on audio chunks.
  - `network`: Wi-Fi management, WebSocket streaming, control JSON receive.
  - `profile_lookup` (event-driven): NFC-triggered profile fetch.
  - `schedule` (event-driven): alarm/timer check and dispatch.
  - `beep` (event-driven): queued beep playback for feedback tones.
- Real-time split: `audio_capture` + `wakeword` on one core (real-time audio); `network` + `uiTask` on the other core (communication + state).

**Key insight:** Real-time audio work (capture + inference) is separated from network and UI work. This prevents Wi-Fi jitter from affecting audio stability.
